Creación dinámica de la información

Introducción al contenido web dinámico•Un servidor web estático se limita a almacenar archivos y enviarlos al navegador cuando son solicitados.•Se presenta siempre el mismo contenido a distintos usuarios.•Poco después de surgir la web, se hizo obvio la necesidad de generación dinámica.•Este contenido se genera mediante programas que se ejecutan en el servidor.
•La generación dinámica de contenido web ha evolucionado a lo largo de varias décadas, en varias etapas:•Programas externos ejecutados por el servidor.•Lenguajes embebidos en páginas HTML.•Servidores de aplicaciones.•Frameworks y servicios web modernos.

Web dinámica•El contenido se genera mediante un programa que se ejecuta en el servidor.•El contenido puede variar en cada petición.•Puede consultarse una base de datos.•Puede procesarse información enviada por el usuario.•Este modelo es el utilizado por la mayoría de aplicaciones web actuales.•Puede tener páginas dinámicas independientes (poco habitual hoy en día), o un punto de acceso común a todas las solicitudes con sistema de ruta.

Modelo cliente-servidor•Las aplicaciones web modernas se basan en el modelo cliente-servidor.•El cliente es el programa que solicita información al servidor, que sirve para:•Enviar peticiones al servidor.•Recibir las respuestas.•Mostrar la información al usuario.•Es básicamente el agente de usuario.•El servidor es el sistema encargado de procesar las peticiones:•Recibe peticiones HTTP.•Ejecuta programas de servidor.•Genera la respuesta correspondiente.

Flujo de una petición web•Cuando un usuario interactúa con una aplicación web se produce una secuencia de operaciones:1.El usuario realiza una acción en el navegador.2.El navegador envía una petición HTTP al servidor.3.El servidor recibe la petición.4.Se ejecuta el programa que corresponda del servidor.5.El programa puede consultar una base de datos.6.El servidor genera la respuesta.7.El navegador muestra el resultado al usuario.•Este proceso suele completarse en milisegundos.•Los servidores web están diseñados para gestionar miles de peticiones simultáneas.
Procesamiento en el servidor•Cuando el servidor recibe una petición HTTP debe procesarla mediante con un programa que implementa la lógica de la aplicación.•El procesamiento típico incluye varias operaciones:•Interpretar la petición recibida.•Determinar qué recurso se solicita.•Ejecutar la lógica de la aplicación.•Acceder a una base de datos si es necesario.•Generar la respuesta HTTP.•El servidor puede ejecutar diferentes programas dependiendo de la ruta solicitada, que apuntará normalmente a una función o controlador específico dentro de la aplicación:•/productos•/usuarios•/pedido/123
Aplicaciones dinámicas•La mayoría de los servicios digitales actuales utilizan generación dinámica de contenido.•En estos sistemas el servidor construye la respuesta en función de:•Los datos almacenados.•Las acciones del usuario.•El estado actual de la aplicación.•Ejemplos habituales:•Tiendas en línea que muestran catálogos de productos.•Redes sociales que muestran publicaciones personalizadas.•Sistemas bancarios que muestran movimientos de cuenta.•Plataformas educativas que gestionan cursos y evaluaciones.•En todos estos casos el contenido no existe previamente como archivo HTML, sino que se genera en el momento de la petición.
Componentes de una aplicación web•Una aplicación web dinámica está formada por varios componentes que trabajan de forma coordinada.•Componentes principales:•Cliente web: el programa que interactúa con el usuario.•Servidor web: recibe las peticiones HTTP y las dirige a la aplicación.•Aplicación del servidor: contiene la lógica del negocio.•Base de datos: información persistente.

Ejemplo de servidor•Para una aplicación que muestre una lista de productos en una tienda en línea, cuando el usuario accede a la página correspondiente se produce el siguiente proceso:•El navegador solicita la ruta /productos.•El servidor recibe la petición HTTP.•El programa backend consulta la base de datos.•Se recupera la lista de productos.•El servidor genera una página HTML con la información.•El navegador recibe el documento generado y lo muestra al usuario.•Para el usuario parece una página normal, pero en realidad ha sido construida dinámicamente en el servidor.
Usos de la caché web•El impacto en rendimiento de generar las páginas dinámicamente dependerá de las acciones concretas, por lo que muchas veces se puede cachear parte de un documento HTML generado dinámicamente, o todo.•Se pueden cachear por separado:•Datos extraídos de bases de datos.•El resultado de procesar esos datos de bases de datos.•Imágenes creadas a partir de otras.•La utilidad de cachear todo el documento HTML se maximiza en sistemas de CDN en la frontera de las redes como Cloudflare.

Servidores web
Qué es un servidor web•Un servidor web es un programa informático, y por extensión la parte física que lo ejecuta, diseñado para procesar, almacenar y entregar contenido basándose en el protocolo HTTP/HTTPS.•Puede usarse para transferir imágenes, vídeos, o simple texto en distintos formatos como HTML, JSON o XML, pregenerados o generados en el momento de la transferencia.•Actualmente el servidor web suele trabajar junto con una aplicación que genera el contenido.
Imagen: https://www.trevenque.es/blog/posts-cloud/que-es-un-servidor-web-funcionamiento-y-tipos/
Funciones principales•Sirve archivos estáticos: HTML, CSS, JS, imágenes.•Enruta peticiones hacia la aplicación correspondiente.•Gestiona conexiones múltiples y simultáneas de forma eficiente.•Aplica seguridad: TLS/HTTPS, cabeceras de seguridad, control de acceso.•Comprime respuestas (GZIP, Brotli) para reducir el ancho de banda.•Cachea respuestas frecuentes para mejorar el rendimiento.•Registra todas las peticiones y errores en registros.

Ciclo de vida de una petición1.El servidor acepta la conexión TCP del cliente.2.Lee los octetos de la petición HTTP.3.Interpreta método, URL, cabeceras y cuerpo.4.Decide qué manejador o archivo le corresponde al tipo de solicitud.5.Busca un archivo o invoca otra aplicación.6.Construye y envía la respuesta HTTP.7.Escribe en el registro: IP, URL, código, tiempo…8.Cierra o mantiene viva la conexión (keep-alive).
•Un ejemplo más concreto:•El cliente abre una conexión TCP al puerto 80 (HTTP) o 443 (HTTPS).•Envía una petición:     GET /index.html HTTP/1.1•El servidor procesa la petición y devuelve una respuesta con código de estado:    200 OK•La conexión se cierra o se reutiliza (keep-alive / multiplexación de HTTP/2) según esté programado el servidor.
Servidores web más utilizados•Existen numerosos servidores web disponibles para diferentes entornos y necesidades.•Los dos servidores gratuitos más extendidos en sistemas Unix y Linux son:•Apache HTTP Server.•NGINX.•Características comunes:•Alto rendimiento.•Gran estabilidad.•Amplia comunidad de usuarios.•Configuración flexible.•Pero hay otros como IIS o LiteSpeed.
•Fue creado en 1995, y es uno de los proyectos de código abierto más influyentes de la historia.•Ofrece flexibilidad y ecosistema maduro, pero consume más memoria bajo alta concurrencia.•Su arquitectura está basada en módulos que se cargan según se se quiera.•Modelo de procesamiento: un hilo o proceso por conexión (MPM prefork/worker/event).•Se configura mediante ficheros .htaccess por directorio, lo cual es muy flexible.•Ideal cuando se necesita configuración granular por sitio o directorio.•Tiene penalización de rendimiento.
Apache HTTP Server
Modos de funcionamiento de Apache (1/2)•CGI (Common Gateway Interface)•Apache ejecuta un nuevo proceso del programa por cada petición HTTP.•El programa genera el HTML dinámicamente.•El proceso finaliza al terminar la petición.•Este sistema tiene limitaciones:•Alto consumo de recursos: los procesos son costosos de generar frente a los hilos.•Lo cual provoca un bajo rendimiento en sistemas con muchas peticiones.
•FastCGI•Los programas se ejecutan como procesos persistentes.•Un mismo proceso puede gestionar muchas peticiones.•Apache se comunica con esos procesos mediante conectores (sockets).•Esto permite reducir la creación continua de procesos.
Modos de funcionamiento de Apache (2/2)•PHP-FPM (FastCGI Process Manager)•Utiliza el protocolo FastCGI para ejecutar PHP, manteniendo un fondo de procesos PHP activos.•Apache o NGINX envían las peticiones a ese fondo.•Ventajas:•Alto rendimiento.•Gestión eficiente de procesos.•Escalabilidad.
•Módulos embebidos (mod_php)•El intérprete PHP se carga dentro del proceso de Apache.•Cada proceso del servidor puede ejecutar código PHP directamente.•Ventaja: simplicidad de configuración.•Limitación: alto consumo de memoria.
•Proxi hacia servidores de aplicaciones•Apache actúa como proxi inverso: recibe peticiones y las reenvía.•En este caso, las peticiones se reenvían a una aplicación externa.
NGINX (1/2)•Fue creado en 2004 para resolver el problema C10K (10 000 conexiones simultáneas).•Arquitectura asíncrona y dirigida por eventos: un proceso gestiona miles de conexiones.•Extremadamente eficiente sirviendo contenido estático y actuando de proxy inverso.•Configuración declarativa, compacta y fácil de auditar.•Muy usado como capa frontal que distribuye tráfico hacia múltiples servidores.•NGINX es utilizado con frecuencia en entornos de alto tráfico, como:•Plataformas de vídeo.•Redes sociales.•Servicios en la nube.

NGINX (2/2)•Puede funcionar en distintos modos según el papel que desempeñe dentro de la arquitectura del sistema.•Los modos de operación habituales son:•Servidor web de contenido estático.•Proxi inverso.•Balanceador de carga.•Puerta de enlace hacia aplicaciones mediante FastCGI.•Proxi hacia aplicaciones de servidor (Go, Node, etc.).

Apache vs. NGINX•El problema es que en estas dos décadas Apache ha evolucionado y NGINX ya no tiene una clara ventaja de rendimiento.•Además, los sistemas de caché actuales, como las CDN, hacen que cualquiera de los dos principales servidores web rindan igual de bien.•Apache se puede usar en más escenarios porque tiene una mayor variedad de módulos desarrollados.•NGINX es más eficiente para sistemas de alta carga, pero si se usa caché esa ventaja se pierde.

Arquitectura interna de un servidor web•Modelo clásico basado en procesos o hilos (Apache):Petición 1 → Proceso A Petición 2 → Proceso B Petición 3 → Proceso C•Simple pero costoso: cada proceso ocupa memoria aunque espere.
•Modelo basado en eventos (NGINX, Go):Petición 1 ──┐ Petición 2 ──┼──► bucle de eventos → E/S asíncronaPetición 3 ──┘•Un solo proceso gestiona miles de conexiones con bajo consumo.•Go usa gorrutinas (concurrencia ligera), lo que lo hace naturalmente eficiente.
Conexiones concurrentes•Un servidor web moderno puede atender miles o incluso millones de peticiones simultáneamente, mediante el uso de:•Múltiples procesos.•Aislamiento elevado.•Mayor consumo de memoria.•Múltiples hilos de ejecución.•Menor consumo de recursos.•Mayor complejidad en la gestión de concurrencia.•Arquitecturas orientadas a eventos.•Alta eficiencia: un hilo, múltiples clientes simultáneos.•Muy utilizadas en servidores modernos.

Potenciales cuellos de botella•Los recursos de un servidor son limitados. Si recibe más conexiones de las que puede gestionar, dejará de prestar el servicio adecuadamente.•Eso se puede dar por varios frentes:•Excesivo uso de CPU procesando código de aplicaciones para generar las respuestas.•Excesivo uso de la B. DD., lo cual puede consumir igualmente mucha CPU o E/S.•Excesiva lectura de archivos. Suelen ser de unos pocos kilobytes, lo cual puede estresar incluso al funcionamiento de una SSD rápida PCIe 4.0 si hay cientos o miles de usuarios.•El propio servidor consume RAM para recibir una solicitud, analizar y hacer lo que deba con ella.
Potenciales soluciones a los cuellos de botella•Para reducir el consumo de CPU se suele cachear todo lo que se pueda cachear, como por ejemplo las consultas a bases de datos.•Si no hay suficientes E/S y sí mucha RAM, se puede hacer un volcado de la caché.•Por norma general, solo se debe procesar y ejecutar lo justo y necesario. •En BB. DD., se pueden paginar los resultados con WHERE, LIMIT y OFFSET, y devolver solo las columnas necesarias.•En lenguajes con punteros, hay que aprovecharse de ellos al máximo.•Si hay limitación de ancho de banda del servidor, y aunque no la haya, se recomienda usar compresión de datos (GZIP, Brotli).•Si el servidor se queda sin capacidad de cómputo, se puede escalar horizontalmente: añadir más instancias del servidor detrás de un balanceador de carga (NGINX, HAProxy).•Antes de aplicar cualquier solución hay que medir. Los registros, el comando EXPLAIN en SQL y herramientas de perfilado pueden indicar dónde está realmente un posible problema de un servidor web.
Archivos de registro del servidor•Los servidores web mantienen archivos de registro que almacenan información sobre su funcionamiento.•Estos registros permiten analizar el comportamiento del sistema.•Tipos de registros habituales:•Registro de accesos.•Registro de errores.•Registro de eventos del sistema.•Los registros son fundamentales para:•Diagnosticar problemas.•Analizar el tráfico del sistema.•Detectar posibles incidentes de seguridad.
•Línea de registro de ejemplo:192.168.1.1 - ana [10/Mar/2025:14:32:01 +0100] "GET /perfil HTTP/1.1" 200 3421 "https://ejemplo.com" "Mozilla/5.0 ..."•Información que puede registrarse:•Dirección IP del cliente.•Fecha y hora de la petición.•Recurso solicitado.•Código de estado HTTP.•Tamaño de la respuesta enviada.•Como siempre, hay que tener cuidado con las implicaciones legales de la RGPD: una dirección IP es un dato personal.

Arquitecturas de aplicaciones web
Problema de la organización del código•A medida que una aplicación web crece, el software se vuelve más complejo.•Sin estructura, se dan los problemas de la programación no estructura:•Código difícil de entender.•Dificultad para modificar funcionalidades.•Alto riesgo de introducir errores.•Escasa reutilización de componentes.•En aplicaciones pequeñas estos problemas pueden ser manejables.•En aplicaciones reales con miles de líneas de código es necesario organizar el software de forma estructurada.•Las arquitecturas de software permiten dividir la aplicación en componentes con responsabilidades bien definidas.

Separación de responsabilidades•Uno de los principios fundamentales del diseño de software es la separación de responsabilidades: cada componente  debe encargarse de una función específica.•El código resulta más fácil de entender.•Las modificaciones se realizan con menor riesgo.•Es posible reutilizar componentes.•Facilita el trabajo en equipo.•En aplicaciones web suele distinguirse entre tres tipos de responsabilidades:•Interfaz de usuario.•Lógica de negocio.•Acceso a datos.•Separar estas funciones permite construir sistemas más robustos y mantenibles:•Cambiar la interfaz no afecta a la lógica•Cambiar la BBDD no afecta a la presentación•Cada capa puede testearse de forma independiente•El equipo puede dividirse el trabajo por capas
Arquitectura en capas•En este modelo el sistema se divide en una jerarquía de niveles.•Capas habituales en una aplicación web:•Capa de presentación.•Capa de lógica de negocio.•Capa de acceso a datos.•Cada capa solo habla con la capa adyacente:•La capa superior no sabe cómo almacena los datos la inferior.•La capa inferior no sabe cómo los muestra  la superior

Modelo-vista-controlador (MVC)•Este patrón de arquitectura divide la aplicación en tres componentes principales:•Modelo: gestiona los datos y la lógica de negocio.•Vista: representa la información mostrada al usuario.•Controlador: gestiona las peticiones y coordina el flujo de la aplicación.•Esta separación permite desarrollar sistemas donde:•La lógica de negocio está claramente aislada.•La interfaz puede modificarse sin afectar al resto del sistema.•El código resulta más fácil de mantener.•El patrón MVC ha sido ampliamente utilizado en entornos de trabajo web y aplicaciones empresariales.
Componentes de MVC•Modelo:•Representa los datos de la aplicación.•Contiene las reglas de validación y negocio.•Se comunica con la base de datos.•No sabe nada de HTTP ni de HTML.•Vista:•Plantillas HTML que muestran los datos.•No contiene lógica de negocio.•Recibe datos del controlador y los presenta.•Controlador:•Recibe la petición HTTP.•Invoca el modelo para obtener o modificar datos.•Selecciona la vista adecuada y le pasa los datos.•Devuelve la respuesta al cliente.
Flujo de control en MVC•El modelo MVC define también cómo circula la información dentro de la aplicación.•Flujo típico de procesamiento:•El usuario realiza una acción en la interfaz.•La petición llega al controlador.•El controlador interpreta la solicitud.•El controlador consulta o modifica el modelo.•El modelo accede a los datos necesarios.•El controlador selecciona la vista adecuada.•La vista genera el HTML con los datos proporcionados.•El controlador envía el HTML.•Este flujo permite mantener una clara separación entre lógica, datos e interfaz.

Modelo-vista-presentador (MVP)•Variante de MVC pensada para interfaces de usuario más complejas:•Modelo: igual que en MVC, gestiona datos y lógica.•Vista: más pasiva, ya que no llama al modelo directamente.•Presentador: intermediario que gestiona toda la lógica de presentación.•En MVC, la vista puede observar directamente al modelo.•En MVP, la vista solo habla con el presentador, nunca con el modelo.
